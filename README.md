# Anomaly & Fraud Detection Platform

A production-*style* anomaly and fraud detection platform in Python, built to demonstrate
MLOps practices end-to-end: synthetic data generation, three comparable models behind a
common interface, a saved preprocessing pipeline, honest evaluation under class imbalance,
a FastAPI serving layer with measured latency, a versioned model registry, gated automated
retraining, drift monitoring, a real test suite, and a Streamlit/Plotly dashboard.

> ## ⚠️ The data is 100% SYNTHETIC
> Every transaction is generated by [`fraud_platform/data/generate.py`](fraud_platform/data/generate.py).
> No real cardholder, bank, or financial data is used anywhere. Because the data is
> synthetic, the fraud signal is partly known by construction, so **metrics here are not
> evidence of real-world fraud-detection performance** — they demonstrate that the pipeline,
> models, and evaluation work correctly. See [What the build honestly supports](#what-the-build-honestly-supports-cv-accuracy).

Runs entirely on a laptop. No GPU, no paid cloud, no external API keys.

## Architecture

```
fraud_platform/
├── config.py                  # paths, seeds, schema, thresholds (single source of truth)
├── data/
│   ├── generate.py            # synthetic transaction generator (CLI)
│   └── loader.py              # load/split/hash helpers
├── pipeline/features.py       # feature engineering + sklearn ColumnTransformer Pipeline
├── models/
│   ├── base.py                # AnomalyModel ABC — the common interface
│   ├── iforest.py             # Isolation Forest (unsupervised)
│   ├── xgb.py                 # XGBoost (supervised)
│   ├── autoencoder.py         # PyTorch autoencoder (reconstruction error)
│   └── __init__.py            # name -> class factory (swappable models)
├── evaluation/evaluate.py     # precision/recall/F1/ROC-AUC/PR-AUC + comparison table
├── registry/registry.py       # local versioned model registry + champion pointer
├── monitoring/drift.py        # PSI + KS drift detection
├── serving/api.py             # FastAPI /predict with per-request latency logging
├── train.py                   # train all 3, evaluate, register
└── retrain.py                 # retrain -> evaluate -> gated promotion
dashboard/app.py               # Streamlit + Plotly (EDA, comparison, live scoring, drift)
tests/                         # pytest suite (45 tests)
requirements.txt, Dockerfile
```

**Data flow:** `generate` → `Pipeline` (engineer + impute/scale/encode) → model `.fit` →
`evaluate` on held-out test → `registry.register` → `serving` loads champion → `/predict`.

The fitted preprocessing **Pipeline is saved inside every model artifact**, so inference
applies the exact same transforms seen at training time. The three models are interchangeable
behind the `AnomalyModel` interface — `score()` always returns a value where *higher = more
anomalous*, regardless of whether the model is supervised or unsupervised.

## Quick start

> On macOS the interpreter is usually `python3` (there's no bare `python`). Use `python3`
> below, or add an alias: `alias python=python3`.

```bash
pip install -r requirements.txt           # see "macOS note" below if XGBoost fails to load

python3 -m fraud_platform.data.generate   # (optional) write a CSV; train.py auto-generates too
python3 -m fraud_platform.train           # generate (if needed) + train all 3 + evaluate + register
python3 -m pytest -q                      # run the test suite

python3 -m uvicorn fraud_platform.serving.api:app   # serve the champion on http://localhost:8000
python3 -m streamlit run dashboard/app.py           # dashboard (run the API first for live scoring)
```

### Score a transaction
```bash
curl -X POST http://localhost:8000/predict -H "Content-Type: application/json" -d '{
  "amount": 920.5, "account_age_days": 45, "num_tx_last_24h": 11,
  "avg_amount_last_30d": 80.0, "distance_from_home_km": 240.0, "hour": 3,
  "merchant_category": "electronics", "transaction_type": "transfer",
  "device_type": "web", "country": "NG"}'
```
Response includes `is_fraud`, `anomaly_score`, `inference_ms`, and `within_latency_budget`.

### Docker
```bash
docker build -t fraud-platform .   # trains the models during build
docker run -p 8000:8000 fraud-platform
```

### macOS note (OpenMP)
PyTorch and XGBoost each ship their own OpenMP runtime; running both multithreaded can
segfault. The package pins `OMP_NUM_THREADS=1` at import in
[`fraud_platform/__init__.py`](fraud_platform/__init__.py) to avoid this. You may also need
`brew install libomp` for XGBoost. On Linux/Docker, `libgomp1` is installed and you can drop
the thread pin for full parallelism.

## Models

| Model | Type | How it scores | Uses labels? |
|-------|------|---------------|--------------|
| Isolation Forest | unsupervised | negated `score_samples` (isolation depth) | no |
| XGBoost | supervised | predicted fraud probability; `scale_pos_weight` for imbalance | yes |
| Autoencoder (PyTorch) | unsupervised | per-row reconstruction MSE; trained on legit rows only | only to pick legit rows |

Each model fits its own preprocessing pipeline and sets a decision threshold by maximising
F1 on a validation split. Add a new model by implementing `AnomalyModel` and registering it
in `models/__init__.py`.

## Results (measured, synthetic data)

Run: `python -m fraud_platform.train --rows 50000 --fraud-frac 0.01`
(50,000 transactions, 1% fraud, stratified 60/20/20 train/val/test, seed 42).
Test set = 10,000 rows containing 100 fraud cases.

| model | precision | recall | F1 | ROC-AUC | **PR-AUC** |
|-------|-----------|--------|------|---------|------------|
| isolation_forest | 0.0488 | 0.2000 | 0.0784 | 0.7669 | 0.0590 |
| **xgboost** | 0.8472 | 0.6100 | 0.7093 | 0.9429 | **0.6712** |
| autoencoder | 0.4783 | 0.3300 | 0.3905 | 0.8205 | 0.2766 |

**Reading these honestly:**
- The baseline PR-AUC for a random model equals the fraud rate, **0.01**. XGBoost's 0.67 is
  far above that; Isolation Forest's 0.059 is better than random but weak.
- **ROC-AUC looks flattering** (all ≥ 0.77) precisely because of class imbalance — this is
  why PR-AUC and recall are the metrics to trust here.
- **Recall is the honest story:** even the best model (XGBoost) catches **61%** of fraud at
  this threshold, missing 39%. Raising recall would cost precision. This trade-off is real
  and is exactly why the synthetic generator includes ~45% "stealth" fraud that looks like
  normal activity — without it, every model trivially scored ~1.0, which would be a
  meaningless (and misleading) result.
- Supervised (XGBoost) clearly beats the unsupervised models on this data, as expected when
  labels are available.

## Latency (measured)

Single-record scoring, 500 requests after warm-up, on a laptop (Apple Silicon, CPU only):

| path | mean | p95 | p99 |
|------|------|-----|-----|
| XGBoost `model.score` (incl. preprocessing) | 2.4 ms | 2.5 ms | 2.6 ms |
| Autoencoder `model.score` | 2.3 ms | 2.4 ms | 2.5 ms |
| Isolation Forest `model.score` | 6.4 ms | 6.6 ms | 6.7 ms |
| **End-to-end `/predict`** (XGBoost, incl. HTTP) | **3.6 ms** | 4.0 ms | 4.6 ms |

**Verdict: yes, the `/predict` latency claim of "< 100ms" holds** — by a wide margin (~3.6ms
end-to-end), single-request, single-threaded, on a laptop. The API logs `inference_ms` on
every request and returns it in the response, so this is verifiable, not assumed. Note these
numbers are for one request at a time; sustained throughput under concurrency was not
benchmarked.

## Model registry & versioning

Home-grown, dependency-free, on disk under `registry_store/`:

```
registry_store/
├── index.json                 # {model_type: {"champion": <version>}}
└── <model_type>/v<N>/
    ├── model.pkl | model.pt    # model + its fitted preprocessing pipeline
    └── metadata.json           # version, utc_timestamp, model_type, metrics, data_hash
```

Each `register()` writes a new **immutable** version; the champion pointer says which version
serving loads. Versions are never overwritten, so you keep full history and lineage
(`data_hash` ties a model to the exact training data).

**Why not MLflow?** In production MLflow (or similar) would be the right choice: a tracking
server, experiment comparison UI, a backing artifact store (S3/GCS), and model-stage
transitions (Staging → Production) with access control. This home-grown registry is
deliberately minimal so the structure is transparent and runs anywhere with zero setup — it
covers versioning, metadata, and a champion pointer, but not MLflow's UI, lineage graph, or
multi-user concerns.

## Automated retraining (gated promotion)

```bash
python3 -m fraud_platform.retrain                       # retrain all on default data
python3 -m fraud_platform.retrain --data new_batch.csv  # retrain on a fresh batch
python3 -m fraud_platform.retrain --model xgboost       # one model type
```

**What triggers a retrain (accurate description):** this is a **manual command**, designed to
be run **on a schedule (e.g. a nightly cron job / CI workflow)** or by hand when a new
labelled batch arrives or drift is flagged. It is **not** event-driven / streaming
auto-retraining — nothing watches the data and fires it for you. Example cron:

```cron
0 2 * * *  cd /app && python -m fraud_platform.retrain
```

**Promotion gate:** a candidate is always registered as a new version, but it only becomes
champion if its **PR-AUC** on the held-out test set is *strictly greater* than the current
champion's. A worse model can never silently replace a good one. (Metric configurable via
`--metric`; PR-AUC chosen because it's the right signal under heavy class imbalance.)

## Drift detection

```python
from fraud_platform.monitoring.drift import compute_drift
result = compute_drift(reference_df, current_df)   # per-feature PSI + KS, overall flag
```

Two complementary per-feature signals:
- **PSI (Population Stability Index)** — effect size of distribution shift. Bands:
  `< 0.10` none · `0.10–0.25` moderate · `≥ 0.25` significant.
- **KS test** — two-sample Kolmogorov–Smirnov; p-value `< 0.05` rejects "same distribution".
  KS over-fires on large samples, which is why PSI's effect-size view is paired with it.

A feature is flagged if `PSI ≥ 0.25` **or** `KS p < 0.05`; the batch drifts if any feature is
flagged. The dashboard's Drift tab lets you inject a synthetic shift and watch features flag.

## Trade-offs made for laptop-runnability

- **Synthetic data** with deliberate fraud/legit overlap (incl. 45% "stealth" fraud) so the
  task is non-trivial and models differ — instead of a real dataset (privacy, size, access).
- **Single-threaded OpenMP** on macOS to avoid the torch/XGBoost OpenMP segfault — fine at
  this data size; relax on Linux for parallelism.
- **Home-grown registry** over MLflow — zero setup and transparent, at the cost of a UI/lineage.
- **Training inside the Docker image** for a one-command demo — a real deployment would ship a
  registry built in CI and mounted, not trained at build time.
